# Swarm Review — 2026-04-19

Five parallel agents audited the codebase across architecture, code quality,
security, UX coherence, and prompt engineering. This is the consolidated
hit list, separating what was fixed this session from what's deferred.

## Deferred — needs a dedicated session

### Security

- **No authentication anywhere.** Admin API is wide open on localhost. Blocks
  multi-tenant deploy. Needs a proper design pass (session cookies? JWT?
  per-tenant API key? auth-via-magic-link?). CORS is currently `origin: true`
  which amplifies the blast radius.
- **Arbitrary code execution during ingest.** `buildProject` runs
  `npm install && npm run build` on uploaded `package.json` scripts. Ultimate
  fix is process-level sandboxing (firecracker / container / nsjail). Interim
  mitigations worth considering: ignore `preinstall`/`postinstall` scripts,
  run with `--ignore-scripts`, enforce a kernel-level CPU/memory cap,
  network-isolate the build.
- **ZIP symlink handling.** `unzipper` doesn't reject symlink entries. An
  attacker could craft a ZIP that writes (or reads) outside the extract dir
  via a symlink. Add `lstat` check; reject `isSymbolicLink()` entries.
- **Snapshot content hygiene.** `.eval/cases/*/snapshot.json` is committed
  with extracted source HTML. Spot check found no secrets but a customer's
  site could contain embedded API keys, email addresses, etc. Consider
  scrubbing or moving `.eval/` out of git once real customer ingestions
  start.

### Architecture

- **`hp-app` + `hp-server` cutover.** The old upload/analyze/deploy app still
  runs outside the workspace. Keep them as a parallel path for now (they do
  something `apps/api` doesn't yet), but plan an explicit retirement when
  the admin UI absorbs their flows. Mark them deprecated in their READMEs.
- **`TenantStore.rebuild()` semantics for Payload.** Currently spawns
  `pnpm run build` in a tenant dir. When `PayloadTenantStore` lands, rebuild
  will have to enqueue a job / hit a webhook instead. The interface is fine
  (`Promise<RebuildResult>`) but implementations will diverge sharply.
  Worth designing the build-orchestration side of the platform before Payload
  integration starts.

### Code quality

- **`rewriteRelativeRefs` regex fragility** (fossilize.ts). Assumes quoted
  attributes + single-line attributes. Unquoted `<img src=foo alt=bar>` and
  multiline attributes slip through. Low urgency (well-built sites always
  quote) but worth replacing with a proper HTML parser (cheerio) when touched
  next.
- **`background-image` URL regex in apply-edits.ts** is greedy and mishandles
  `url("data:...")` with quotes inside. Switch to a tokenizer when we add
  CSS-embedded edit support in earnest.
- **`file-store.ts` tenant-data.ts JSON extraction via regex.** If the
  generator ever changes the emission format (extra comments, different
  trailing whitespace), parsing silently falls back to defaults. Move to
  a small helper that either parses via `esbuild` or writes the data as a
  `.json` sidecar alongside `tenant-data.ts`.
- **Type casts without runtime validation** across many packages: `JSON.parse`
  → `as CarveMap`, `as EditsMap`, `as TenantData`. One malformed file and a
  cascade of misbehavior. Add a small Zod-based validator at module
  boundaries.
- **PageEditor dirty-state closure pattern** (`mergedValueFor`) works but is
  cognitively heavy. When we add more field types (collections, repeaters,
  media picker) this should move to a proper form state library (`react-hook-form`
  or similar).

### UX

Ranked by "would the user complain first":

1. **No single entry point** — new user has to learn `pnpm ingest → eval → generate → admin` before anything works. Hide the pipeline behind one "Upload ZIP" button in admin.
2. **Admin is write-only, not a CMS.** No media picker, no color picker, no
   page composer, no "add a blog post" flow. The promise "CMS-editable" is
   only half-delivered.
3. **Silent failures** — rebuild errors truncated at 600 chars, carve-0-edits
   shown with no remediation, apply-edits skip warnings invisible to the
   non-technical admin. Needs proper error surfaces.
4. **No preview / no publish separation.** Save writes straight through;
   there's no draft state. Should be explicit.
5. **CLI / admin feature mismatch** — power users get more from CLI than the
   UI exposes. Either surface it or hide it.

### Prompt engineering ROI (ranked)

1. **`pixel-block-generator`** — `props: Record<string, unknown>` is too
   loose. Leads to component-name collisions when the same block appears on
   multiple pages with different prop shapes. Add schema enforcement + a
   second-pass validation step. Highest-impact tune.
2. **`extract.ts`** (collection entries) — no explicit confidence bias
   stated; oversamples pages (30 at full HTML). Tighten schema enforcement
   inline + reduce sample to top 10 relevant pages.
3. **`carve.ts`** — no rule for "decorative SVG vs content icon," no rule
   for "pure-chrome page returns empty + note." Small prompt additions.
4. **`tokens/extract.ts`** — output schema shown as TypeScript comment;
   Claude can omit categories and `mergeWithDefaults` silently patches.
   Better: ship a JSON example; validate required fields; fail loud on
   incomplete output.
5. **`detect.ts`** — already well-engineered; no urgent tune.

### Agent loop hardening

- `callClaudeJson` retries once on JSON parse failure. Works for syntax
  errors; misses semantic errors (malformed Astro, schema violations,
  selectors that don't resolve). For `pixel-block-generator` specifically,
  a two-pass validate-and-patch loop would prevent the post-hoc
  component-collision warnings. Other skills probably fine with the
  current single-retry.
