Three approaches to autonomous analysis

All use tool use + shared state. The difference is how structured the exploration is.

A

Single autonomous agent — recommended

One Claude session. System prompt defines SOL analyst. We send the file tree + one instruction: "Analyze this project completely." AI calls tools freely until it understands the project, then calls write_analysis(json) to finish.

Tools the AI gets
list_files(dir?)
read_file(path)
search_in_file(path, pattern)
write_analysis(schema) ← terminal tool
SSE stream shows
🔍 Reading js/router.js...
✓ Found hash routing — 7 routes
🔍 Reading js/pages/blog.js...
✓ Blog page: 3 sections found
🔍 Reading js/blog-data.js...
✓ Content: blogPosts array (12 items)
✓ Analysis complete — writing result

Pros

  • AI follows the actual code — no assumptions
  • Handles any project structure
  • SSE maps to real exploration steps
  • Simple to implement

Cons

  • Variable time (email solves this)
  • Token cost scales with project size
  • Need a max-tools budget (40 calls)
B

Autonomous agent + progressive checkpoints

Same as A but after the AI explores, we inject checkpoint messages: "Output everything you know about pages so far as JSON, then continue." This gives us partial results during exploration — the dashboard can start rendering while analysis is still in progress.

Pros

  • Partial results appear live — faster perceived loading
  • Dashboard renders pages while content is still being analyzed

Cons

  • Complex prompt engineering to not interrupt AI flow
  • Partial results need careful merging with final output
C

Multi-turn phases, each with tool use

Keep the current phase structure (Understand → Pages → Content → Tech → Launch) but each phase runs in the same conversation thread and can call tools to read specific files. Full shared state, but phases give predictable SSE labels.

Pros

  • Predictable phases = clear SSE progress
  • Each phase is focused and bounded
  • Easiest migration from current architecture

Cons

  • AI still constrained by our phase structure
  • May miss connections that span phases
  • More API message overhead
My recommendation: A
Start with A — it's the cleanest mental model and maps directly to how a developer actually explores an unfamiliar codebase. The SSE reflects real AI decisions, not our guesses about what files matter. B can be added later as an optimization once A is working. C solves the wrong problem (we already moved past phase-driven architecture).
